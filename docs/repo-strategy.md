# Core repo strategy

## One living trajectory artifact

`trajectory-native` is the **core trajectory layer**:

- central calibration memory
- long-lived evolving artifact
- continuity of reasoning and strategic direction
- accumulated calibration history
- execution residue linkage (GitHub-native)

Avoid excessive repo fragmentation. **Do not create a new repo for every new idea.**

## What this repo preserves

- continuity
- repeated-action identity
- evolving reasoning
- strategic trajectory
- calibration history under timing pressure

## When satellite repos are acceptable

Only when:

- a technical experiment becomes independently useful
- reusable infrastructure emerges
- an execution spike deserves isolation
- a standalone wedge/tool appears

Examples (hypothetical, linked to core thesis):

- `github-context-parser`
- `calibration-graph-engine`
- `trajectory-visualization-layer`
- `execution-residue-indexer`

Satellites must remain **connected to the core trajectory thesis** — not orphan products.

## Founder workflow alignment

This repo functions simultaneously as:

- product (MVP UI)
- thesis (docs/)
- calibration memory (signals, notes, windows)
- execution residue (commits, feed)
- strategic adaptation artifact (trajectory-notes/)

Public calibration and repeated execution in one place.
